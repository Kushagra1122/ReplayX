# ReplayX Demo Runbook

Follow this runbook to show the strongest recruiter-facing ReplayX flow:

- a user reports a bug in Slack
- Slack starts a live ReplayX run
- the dashboard updates live as the orchestrator advances
- the run ends with a postmortem and a reusable incident skill

The best incident for this flow remains the checkout race condition because it is concrete, high-signal, and easy to narrate.

## Demo Modes

ReplayX now supports two demo modes:

1. **Primary mode: Live run**
   - Bug report in Slack
   - Live run starts immediately
   - Dashboard updates live as the orchestrator advances
   - Postmortem and reusable skill appear at the end

2. **Backup mode: Replay**
   - Precomputed artifacts power `/replay/incident-checkout-race-001`
   - Use this only if you want a safety net or if live orchestration is unavailable

For a recruiter demo, use the **live run** as the main path.

## Prerequisites & Setup

Ensure you are in the root directory: `/Users/sourabhkapure/Desktop/ReplayX`

### 1. Environment Configurations
Make sure your `.env` files are in place:
1. Copy the orchestrator environment variables (optional, defaults apply):
   ```bash
   cp .env.example .env
   ```
2. Copy the Slack environment variables (add your credentials):
   ```bash
   cp slack/.env.example slack/.env
   ```
   *Required in `slack/.env`: `SLACK_SIGNING_SECRET`, `SLACK_BOT_TOKEN`, `SLACK_BUGS_CHANNEL_ID`*
   *For live Slack -> dashboard orchestration, set these exactly in `slack/.env`:*
   - `REPLAYX_DASHBOARD_URL=http://localhost:3001`
   - `REPLAYX_ORCHESTRATOR_URL=http://localhost:3001`
   - `REPLAYX_INTERNAL_API_TOKEN=<same token used by dashboard API>`

3. Optional dashboard env:
   If you want the dashboard API to require auth for run creation, export the same token in the shell that launches the dashboard:
   ```bash
   export REPLAYX_INTERNAL_API_TOKEN=<same token used in slack/.env>
   ```

### 2. Install Dependencies
Run from the project root:
```bash
pnpm install
```

## Primary Demo Setup: Live Run

Open **three** terminal tabs from the project root `/Users/sourabhkapure/Desktop/ReplayX`.

### Terminal 1: Start the Target Demo App
Run the broken application that will be targeted by the incident.
```bash
pnpm demo-app
```

### Terminal 2: Start the ReplayX Dashboard
This is the main, judge-facing interactive surface. It now runs behind the custom dashboard server so live runs can update over WebSockets.
```bash
pnpm --dir dashboard dev -- --port 3001
```

### Terminal 3: Start the Slack Intake Service
Required to show the Slack-bot webhook trigger in real time.
```bash
npm start --prefix slack
```
Slack service should log port `3000` and that is expected.

## Primary Demo Flow: Live Run

Use this sequence in the meeting or screen share:

1. **Bug Encounter**
   - Open the **Demo App** from Terminal 2.
   - Show the failing state for the checkout race condition.
   - Keep this short: the point is to make the incident feel real, not to stay in the broken app.
   
2. **Slack Handoff (The Trigger)**
   - Head over to your configured Slack workspace's bugs channel.
   - Mention the ReplayX bot and paste a short bug report, for example:
     - `@ReplayX checkout is overselling stock during concurrent orders`
   - The bot should acknowledge the bug and return a live dashboard handoff URL in the form `/live/<runId>`.
   
3. **Live Orchestrator Run (Dashboard)**
   - Click the link generated by the Slack bot.
   - The live page should open immediately and start updating as the orchestrator advances.
   - Call out the current phase changing in real time.
   - Show the worker fleet appearing once diagnosis has started.
   - Emphasize the winning diagnosis and the selected fix path.
   - Point out that the verification proof is part of the product, not a manual follow-up.
   - Save the “Reusable Skill” callout for the end of the run, when it appears as a final artifact.

## Backup Demo Flow: Replay

If you need a simpler fallback:

1. Open `http://localhost:3001/replay/incident-checkout-race-001`
2. Explain that this is the precomputed judge-safe replay path
3. Walk through diagnosis, fix, proof, postmortem, and skill as a polished artifact story

Use this only if the live path is unavailable or you need a no-risk backup.

## What To Say

Use language like this:

- “Slack is the intake layer.”
- “ReplayX starts a live incident run immediately.”
- “The dashboard updates live as the orchestrator advances through intake, repro, diagnosis, challenger, fix, review, and postmortem.”
- “At the end of the run, ReplayX emits a reusable incident skill.”

Avoid saying things like:

- “It polls a file.”
- “This is just a replay.”
- “It’s basically a chatbot for logs.”

## Preflight Checklist

Run this checklist before the recruiter call:

1. The demo app starts and the seeded bug is still reproducible.
2. The dashboard starts with:
   ```bash
   pnpm --dir dashboard dev -- --port 3001
   ```
3. Slack starts without auth or boot errors.
4. A Slack mention creates a run and returns a `/live/<runId>` handoff.
5. The live dashboard page updates through multiple phases without reload.
6. The reusable skill appears only at the end of the run.
7. `pnpm --dir dashboard build` passes.

## Optional Backup Checklist

Only run this if you want the replay route available as fallback.

1. `pnpm golden-run incidents/checkout-race-condition.json` succeeds.
2. `http://localhost:3001/replay/incident-checkout-race-001` loads cleanly.

## Fallback Plan

If Slack is unavailable during the demo:

1. Keep the dashboard and demo app running.
2. Start a live run manually:
   ```bash
   curl -s -X POST http://localhost:3001/api/replayx/runs \
     -H 'content-type: application/json' \
     --data '{"source":"manual","text":"checkout is overselling stock during concurrent orders"}'
   ```
3. Open the returned `livePath`.

This still demonstrates the same product value: live orchestration, live dashboard updates, and final incident memory.
